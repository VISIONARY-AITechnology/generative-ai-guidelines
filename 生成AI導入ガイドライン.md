## 第1章 目的・適用範囲・用語定義

### 1.1 目的

本ガイドラインは、当社における生成AI（Generative AI）の導入・利用・開発・提供において、安全性の確保、法令遵守、リスク管理を徹底し、AIの便益を最大化しつつ社会的責任を果たすことを目的とする。

> **根拠**: 総務省AI事業者ガイドライン p.2-3「AIの安全安心な活用が促進されるよう、我が国におけるAIガバナンスの統一的な指針を示す」

### 1.2 適用範囲

本ガイドラインは、以下の者に適用される。

| 対象者 | 適用内容 |
|-------|---------|
| 全役職員（正社員・契約社員） | 全条項 |
| 派遣社員 | 業務上利用する場合、全条項準用 |
| 業務委託先 | 契約に本ガイドライン遵守条項を含める |
| インターン・アルバイト | 区分1の利用に限り適用 |

### 1.3 用語定義

| 用語 | 定義 | 根拠 |
|-----|------|------|
| **生成AI（Generative AI）** | 文章、画像、プログラム等を生成できるAIモデルにもとづくAIの総称 | 総務省 p.10 |
| **LLM（大規模言語モデル）** | 大量のテキストデータを学習した大規模な言語モデル。GPT、Claude、Gemini等 | IPA p.14 |
| **AIエージェント** | 自律的にタスクを実行し、外部ツールと連携して目標を達成するAIシステム | 推測（一般的定義） |
| **RAG（検索拡張生成）** | 外部データベースから関連情報を検索し、回答生成に活用する技術 | IPA p.15-16 |
| **ファインチューニング** | 学習済みモデルに対して追加データで再学習を行い、特定分野に特化させる技術 | IPA p.15 |
| **プロンプト** | 生成AIへの入力となる指示文・質問文 | IPA p.16-17 |
| **ハルシネーション** | 生成AIが事実に基づかない情報を生成する現象 | IPA p.41 |
| **AIガバナンス** | AIのリスクを受容可能な水準で管理しつつ、便益を最大化するための技術的・組織的・社会的システムの設計及び運用 | 総務省 p.10 |
| **AI開発者** | AIシステムを開発する事業者 | 総務省 p.5 |
| **AI提供者** | AIシステムをサービスとして提供する事業者 | 総務省 p.5 |
| **AI利用者** | 事業活動においてAIシステム・サービスを利用する事業者 | 総務省 p.5 |

## 第2章 基本原則

本章では、当社における生成AI活用の基本原則を定める。全ての利用・開発・提供活動は、以下の原則に基づいて行われなければならない。

> **根拠**: 総務省AI事業者ガイドライン p.11-21「基本理念」「原則」「共通の指針」

### 2.1 人間中心の原則

- AIは人間の能力を拡張し、人間の幸福（well-being）の追求を支援するものである
- 人間の尊厳及び個人の自律を尊重する
- AIによる意思決定・感情の不当な操作を行わない
- 偽情報・誤情報の生成・拡散を防止する

> **根拠**: 総務省 p.13-15

### 2.2 安全性の確保

- ステークホルダーの生命・身体・財産・精神及び環境に危害を及ぼさない
- AIシステムの信頼性・堅牢性・制御可能性を確保する
- 適切なリスク分析を実施し、対策を講じる

> **根拠**: 総務省 p.15-16

### 2.3 法令遵守

- 個人情報保護法、著作権法その他関連法令を遵守する
- 各AIサービスの利用規約を遵守する
- 国際的な指針・基準を参照し、グローバルスタンダードに準拠する

> **根拠**: 総務省 p.13

### 2.4 説明責任（アカウンタビリティ）

- トレーサビリティを確保し、AIの判断過程を検証可能とする
- ステークホルダーに対して適切な情報を提供する
- 責任者を明示し、問合せに対応する体制を整備する

> **根拠**: 総務省 p.19-21

### 2.5 最小権限の原則

- AIシステム及び関連システムには、業務遂行に必要な最小限の権限のみを付与する
- 定期的に権限の妥当性を見直す

> **根拠**: IPA p.62-63、推測（一般的実務）

### 2.6 データ最小化の原則

- 目的達成に必要な最小限のデータのみを入力・処理する
- 不要なデータは速やかに削除する

> **根拠**: 総務省 p.17-18、推測（一般的実務）

### 2.7 公平性の確保

- 特定の個人・集団への不当なバイアス・差別を排除する
- AIモデルの各構成技術に含まれるバイアスに配慮する
- 適切なタイミングで人間の判断を介在させる

> **根拠**: 総務省 p.16-17

### 2.8 透明性の確保

- AIの判断に係る検証可能性を確保する
- 入出力等のログを記録・保存する
- 関連するステークホルダーに対して必要な情報を提供する

> **根拠**: 総務省 p.18-20

## 第4章 利用区分

当社における生成AIの利用は、以下の3区分に分類される。区分ごとに適用されるルール・審査レベルが異なる。

> **根拠**: 総務省 p.30-39（AI開発者/提供者/利用者の分類）、IPA 全体

### 4.1 区分1：社内業務での利用

#### 4.1.1 定義
役職員が自身の業務効率化のために生成AIを利用する場合。

#### 4.1.2 対象業務の例
- 文章作成・要約・校正
- コーディング補助（コード生成、デバッグ支援）
- アイデア出し・ブレインストーミング
- 情報検索・調査補助
- 翻訳・多言語対応

#### 4.1.3 適用ルール
- 許可ツールリストに掲載されたサービスのみ使用可
- 第5章「データ分類と取り扱い」に従った入力制限
- 事前申請：**不要**（許可ツール利用時）
- 利用ログ：**自動記録**（ツール側機能）

### 4.2 区分2：顧客データを扱う利用

#### 4.2.1 定義
受託開発等において、顧客から預かったデータを生成AIに入力・処理する場合。

#### 4.2.2 対象業務の例
- 顧客提供データの分析・要約
- 顧客システムのテストデータ生成
- 顧客ドキュメントの翻訳・変換
- RAGによる顧客ナレッジ検索システムの構築

#### 4.2.3 適用ルール
- 顧客との契約でAI利用が許可されていることを確認
- 顧客機密データの入力可否を事前確認
- オプトアウト（学習無効化）設定の確認
- 事前申請：**必要**（AI管理者への申請）
- 利用ログ：**必須保存**（最低1年間）

### 4.3 区分3：顧客へ提供するAI機能・AIエージェント

#### 4.3.1 定義
受託開発・SES等において、顧客向けにAI機能を組み込んだシステムを開発・提供する場合。

#### 4.3.2 対象業務の例
- AI機能を組み込んだWebアプリケーションの開発
- AIエージェント（自動応答Bot、業務自動化等）の開発
- RAGシステムの構築・納品
- AI活用コンサルティング

#### 4.3.3 適用ルール
- AI審査会議での事前審査必須
- 顧客との責任分界の明確化（第10章参照）
- セキュリティ要件・品質要件の適用（第8章・第9章参照）
- 事前申請：**AI審査会議審議**
- 運用後監視：プロジェクト完了後も一定期間のフォロー

### 4.4 区分判定フロー

```
Q1. 顧客へAI機能を提供するか？
    ├─ Yes → 区分3
    └─ No → Q2へ

Q2. 顧客データを入力・処理するか？
    ├─ Yes → 区分2
    └─ No → 区分1
```

## 第5章 データ分類と取り扱い

### 5.1 データ分類基準

| 分類 | 定義 | 例 |
|-----|------|-----|
| **公開情報** | 既に公開されている情報、公開しても問題ない情報 | 公開Webサイトの情報、プレスリリース |
| **社外秘** | 社外への開示を制限する情報 | 社内マニュアル、議事録、企画書 |
| **機密情報** | 漏洩により重大な損害が生じる情報 | 経営情報、未発表製品情報、契約書 |
| **個人情報** | 特定の個人を識別できる情報 | 氏名、住所、メールアドレス、顔写真 |
| **特定個人情報** | マイナンバーを含む個人情報 | マイナンバー、基礎年金番号 |
| **顧客機密** | 顧客から預かった機密情報 | 顧客の技術情報、顧客の個人情報 |

> **根拠**: IPA p.40-41、総務省 p.17-18

### 5.2 入力可否マトリクス

| データ分類 | 許可ツール（オプトアウト済） | 未検証ツール | 自社構築環境 |
|-----------|---------------------------|-------------|-------------|
| 公開情報 | ○ | △（要確認） | ○ |
| 社外秘 | △（条件付き可※1） | × | ○ |
| 機密情報 | ×（例外申請必要） | × | △（要審査） |
| 個人情報 | ×（匿名化後のみ） | × | △（要審査） |
| 特定個人情報 | × | × | × |
| 顧客機密 | ×（顧客許可時のみ） | × | △（顧客許可時） |

**凡例**: ○=可、△=条件付き可、×=禁止

**※1 条件付き可の条件**:
- オプトアウト（学習無効化）設定が有効であること
- ログが記録される設定であること
- データの一部のみを入力し、文脈から特定されないこと

### 5.3 入力前確認事項

1. **データ分類の確認**: 入力しようとするデータは何に分類されるか
2. **ツール設定の確認**: オプトアウト設定は有効か
3. **必要性の確認**: 目的達成に本当に必要なデータか（データ最小化）
4. **匿名化の検討**: 個人情報は匿名化できないか

## 第6章 禁止事項

以下の行為は明示的に禁止される。違反した場合は就業規則に基づく処分の対象となる場合がある。

> **根拠**: IPA p.38-42

### 6.1 入力に関する禁止事項

| 禁止事項 | 具体例 |
|---------|-------|
| 特定個人情報の入力 | マイナンバー、基礎年金番号 |
| 顧客許可のない顧客データの入力 | 顧客のソースコード、顧客DB情報 |
| 機密情報の無断入力 | 未発表製品情報、M&A情報、経営戦略 |
| 学習許可状態での機密入力 | オプトアウト未設定ツールへの社外秘以上の入力 |
| 著作権侵害コンテンツの入力 | 他者の著作物を無断でプロンプトに含める |

### 6.2 出力に関する禁止事項

| 禁止事項 | 具体例 |
|---------|-------|
| 生成物の無検証での外部公開 | ハルシネーション確認なしでの報告書提出 |
| 生成物を自己の著作物と偽ること | AI生成であることを隠して納品 |
| 有害コンテンツの生成・利用 | 差別的表現、暴力的表現、違法行為の助長 |
| 第三者権利侵害コンテンツの利用 | 類似性確認なしでの商標・意匠の利用 |

### 6.3 学習・共有に関する禁止事項

| 禁止事項 | 具体例 |
|---------|-------|
| 無許可でのファインチューニング | 顧客データを使った無断学習 |
| プロンプト・テンプレートの無断外部共有 | 社内ノウハウを含むプロンプトのSNS投稿 |
| 社内限定ナレッジの外部漏洩 | RAG用社内文書の外部持ち出し |

### 6.4 アカウント・端末に関する禁止事項

| 禁止事項 | 具体例 |
|---------|-------|
| アカウントの貸与・共有 | 自分のIDを他者に使わせる |
| 未承認ツールの利用 | 許可リストにないAIサービスの業務利用 |
| 私用端末での業務利用 | BYODポリシーに反する利用 |

### 6.5 セキュリティに関する禁止事項

| 禁止事項 | 具体例 |
|---------|-------|
| プロンプトインジェクション攻撃の実行・加担 | システムプロンプト抽出の試行 |
| セキュリティ機能の無効化 | ガードレールの迂回 |
| 脆弱性の悪用 | 既知の脆弱性を利用した攻撃 |

## 第7章 リスク評価と事前審査

### 7.1 リスクアセスメント手法

リスク評価は以下の4段階で実施する。

> **根拠**: IPA p.52-60

#### 7.1.1 リスク特定
- 資産の特定（システム、データ、ネットワーク）
- 各資産に紐づくリスクの洗い出し
- OWASP Top 10 for LLMを参照

#### 7.1.2 リスク分析
- 影響度（High/Medium/Low）の評価
- 発生可能性（High/Medium/Low）の評価

#### 7.1.3 リスク評価
- リスクマトリクスによる優先度付け
- 対応/非対応の閾値決定

#### 7.1.4 リスク対応
- 回避・軽減・受容・移転の判断
- 具体的対策の実施

### 7.2 OWASP Top 10 for LLM（参照）

| No. | リスク | 概要 |
|-----|-------|------|
| 1 | プロンプトインジェクション | 巧妙な入力によるLLMの操作 |
| 2 | 安全でない出力ハンドリング | LLM出力の無検証での連携システム送信 |
| 3 | 訓練データの汚染 | 学習データの改ざん |
| 4 | モデルのDoS | 計算リソースの過剰消費 |
| 5 | サプライチェーンの脆弱性 | コンポーネント・サービスの脆弱性 |
| 6 | 機微情報の漏洩 | 応答への意図しない機密データ混入 |
| 7 | 安全でないプラグイン設計 | 入力検証・アクセス制御の不備 |
| 8 | 過剰な代理行為 | 過剰な機能・権限・自律性の付与 |
| 9 | 過度の信頼 | 出力の無批判な受容 |
| 10 | モデルの盗難 | 不正アクセスによるモデル流出 |

> **根拠**: IPA p.55-56

### 7.3 新規ツール導入時の審査基準

| 審査項目 | 確認内容 |
|---------|---------|
| 提供元の信頼性 | 企業規模、実績、セキュリティ認証 |
| データの取り扱い | 学習への利用有無、保管場所、保持期間 |
| セキュリティ機能 | 認証、暗号化、ログ、監査機能 |
| 利用規約 | 禁止事項、責任範囲、準拠法 |
| コスト | 初期費用、運用費用、スケーラビリティ |

### 7.4 AIエージェント設計の審査観点

| 審査項目 | 確認内容 |
|---------|---------|
| 権限設計 | 最小権限原則の適用、不要な権限の排除 |
| 外部接続 | 接続先の洗い出し、認証方式、通信暗号化 |
| 実行制限 | 実行可能なアクションの制限、タイムアウト設定 |
| 監査ログ | 全アクションの記録、改ざん防止 |
| プロンプトインジェクション対策 | 入力検証、ガードレール、防御用プロンプト |
| 緊急停止機能 | 手動停止、自動停止条件 |

> **根拠**: 総務省 p.25-27（高度なAIシステムの指針）、推測（一般的実務）

## 第8章 セキュリティ要件

### 8.1 認証・認可

| 要件 | 内容 | 適用区分 |
|-----|------|---------|
| SSO（シングルサインオン） | 社内ID基盤との連携推奨 | 全区分 |
| MFA（多要素認証） | 区分2・3では必須、区分1は推奨 | 区分2/3必須 |
| 権限の定期レビュー | 四半期ごとの権限棚卸し | 全区分 |

### 8.2 権限管理

- 最小権限原則の適用
- 職務分掌に基づく権限設計
- 特権アカウントの厳格管理

> **根拠**: IPA p.62-63

### 8.3 ログ管理

| ログ種別 | 記録内容 | 保存期間 |
|---------|---------|---------|
| アクセスログ | ユーザーID、日時、接続元IP | 1年間 |
| 操作ログ | 入力プロンプト、出力内容 | 1年間（区分2/3は3年間） |
| エラーログ | エラー内容、発生日時 | 1年間 |
| 監査ログ | 設定変更、権限変更 | 3年間 |

> **根拠**: 総務省 p.18-19、IPA p.64

### 8.4 データ保護

| 対策 | 内容 |
|-----|------|
| 通信暗号化 | TLS 1.2以上 |
| 保管時暗号化 | AES-256等 |
| DLP検討 | 機密データの外部送信検知・遮断（区分2/3で検討） |
| バックアップ | 定期バックアップと復旧テスト |

### 8.5 多層防御

> **根拠**: IPA p.61-64

#### 8.5.1 入口対策
- プロンプト文の検証（禁止文字列のフィルタリング）
- ガードレールの設置（入力ポリシー違反の検知）
- 入力ログの保存

#### 8.5.2 内部対策
- 最小権限でのプロセス実行
- 防御用プロンプトテンプレートの実装
- コンテナ・サンドボックス環境の利用

#### 8.5.3 出口対策
- 出力内容のガードレール検証
- 有害コンテンツフィルタリング
- 出力ログの保存

### 8.6 AIエージェント固有のセキュリティ要件

| 要件 | 内容 |
|-----|------|
| 外部ツール接続の認証 | OAuth2.0、APIキーの安全な管理 |
| 実行範囲の制限 | 許可されたアクションのホワイトリスト化 |
| タイムアウト設定 | 無限ループ防止、最大実行時間の設定 |
| サンドボックス実行 | 本番環境への影響隔離 |
| 監査ログ | 全外部呼び出しの記録 |

### 8.7 RAG固有のセキュリティ要件

| 要件 | 内容 |
|-----|------|
| アクセス権限管理 | ユーザー/ロール別のデータアクセス制限 |
| データ投入時の審査 | 機密レベルの確認、承認フロー |
| 引用元表示 | 出典の明示によるトレーサビリティ確保 |
| 定期棚卸し | 古い情報・重複情報の削除 |

> **根拠**: IPA p.70

## 第9章 品質・安全性要件

### 9.1 ハルシネーション対策

| 対策 | 内容 | 適用区分 |
|-----|------|---------|
| 人間による検証 | 生成物の正確性を人間が確認 | 全区分 |
| RAGの活用 | 信頼できるソースからの情報検索 | 区分2/3 |
| 複数ソースとの照合 | 生成内容を他の情報源で検証 | 全区分 |

> **根拠**: IPA p.41, p.73-74

### 9.2 バイアス対策

| 対策 | 内容 |
|-----|------|
| 出力のモニタリング | 偏りのある出力の検出 |
| 多様な視点の確保 | 複数のプロンプト設計、複数モデルの比較 |
| 人間の判断介在 | 重要な判断には必ず人間の確認を挟む |

> **根拠**: 総務省 p.16-17

### 9.3 Human-in-the-loop

以下の場面では、人間の判断を必須とする。

- 外部公開コンテンツの最終確認
- 顧客への報告・納品物
- 重要な意思決定への活用
- 個人に関する評価・判断

> **根拠**: 総務省 p.17「人間の判断の介在」

### 9.4 再現性の確保

| 対策 | 内容 |
|-----|------|
| プロンプトの記録 | 使用したプロンプトの保存 |
| モデルバージョンの記録 | 使用時点のモデルバージョン記録 |
| 設定の記録 | temperature等のパラメータ記録 |

### 9.5 性能評価（区分3向け）

| 評価項目 | 評価方法 |
|---------|---------|
| 回答の正確性 | テストデータセットでの正答率 |
| バイアスの有無 | 公平性テスト |

| スケーラビリティ | 負荷テスト |

> **根拠**: IPA p.36

## 第10章 法務・知財・契約

### 10.1 著作権

#### 10.1.1 入力時の注意
- 他者の著作物を無断でプロンプトに含めない
- 引用する場合は著作権法の要件を満たす

#### 10.1.2 出力物の取り扱い
- 生成AIの出力は、そのままでは著作物に該当しない場合がある
- 人間の創作的関与がある場合は著作物となりうる
- 既存著作物との類似性を確認する

> **根拠**: IPA p.42

### 10.2 個人情報保護

- 個人情報の入力は原則禁止
- 匿名化処理を行った上で利用
- プライバシーポリシーへのAI利用の明記

> **根拠**: 総務省 p.17-18

### 10.3 秘密保持

- NDA締結済み案件での情報入力可否を確認
- オプトアウト設定の確認
- 契約違反とならないことの確認

### 10.4 責任分界

#### 10.4.1 受託開発における責任分界

```
【当社の責任範囲】
- 契約に定めた仕様どおりのAI機能開発
- 開発段階でのセキュリティ対策
- 納品物のドキュメント整備
- 瑕疵担保責任（契約条件による）

【顧客の責任範囲】
- 運用環境の構築・維持
- 運用段階でのデータ管理
- エンドユーザーへの利用規約整備
- AI出力の最終判断・利用
```

#### 10.4.2 SESにおける責任分界

```
【当社の責任範囲】
- 技術者の労働者派遣法遵守
- 技術者のスキル・コンプライアンス教育
- 本ガイドラインの技術者への周知

【顧客の責任範囲】
- 指揮命令下での業務指示
- 顧客環境でのセキュリティ管理
- 顧客ポリシーの提示
```

#### 10.4.3 コンサルティングにおける責任分界

```
【当社の責任範囲】
- 専門的見地からの助言・提案
- 情報の正確性についての合理的な確認
- 秘密情報の適切な管理

【顧客の責任範囲】
- 助言に基づく最終的な意思決定
- 実装・運用の実行
- 実装結果の検証
```

### 10.5 免責条項テンプレート

```
【AI機能に関する免責事項】
1. AI機能の出力は参考情報であり、最終的な判断は利用者が行うものとします。
2. AI機能の出力の正確性、完全性、適時性について、当社は保証しません。
3. AI機能の利用により生じた損害について、当社の故意または重過失による場合を除き、
   当社は責任を負いません。
4. AI提供者（モデル提供者）の利用規約変更、サービス停止等により
   AI機能が利用できなくなった場合、当社は責任を負いません。
```

> **根拠**: 推測（一般的実務）、法務確認要

## 第11章 運用ルール

### 11.1 利用申請フロー

#### 11.1.1 区分1（社内業務利用）
- 許可ツールリスト内のサービス：**申請不要**
- 新規ツール利用希望：AI管理者に申請

#### 11.1.2 区分2（顧客データ取扱い）

1. 企画段階でAI審査会議に上程

3. 顧客との責任分界合意
4. 承認後、開発開始

> **根拠**: IPA p.46, p.70

### 11.5 モデル更新時の対応

- 必要に応じてテスト実施
- 更新履歴の記録

## 第12章 事故・インシデント対応

### 12.1 インシデント分類

| 分類 | 例 | 対応レベル |
|-----|-----|-----------|
| 重大 | 機密情報漏洩、大規模サービス停止 | 役員報告、顧客報告 |
| 中程度 | 社外秘情報の誤入力、一時的な誤動作 | AI推進責任者報告 |
| 軽微 | ハルシネーションの検出、軽微な誤出力 | AI管理者報告 |

### 12.2 報告フロー

```
[インシデント発見]
    ↓
[発見者] → 直ちにAI管理者へ報告（口頭・チャット可）
    ↓
[AI管理者] → 初期対応の指示、記録開始
    ↓
[重大度判定]
    ├─ 重大 → AI推進責任者・役員へエスカレーション
    ├─ 中程度 → AI推進責任者へ報告
    └─ 軽微 → AI管理者で対応完結
    ↓
[初動対応] → 被害拡大防止措置
    ↓
[原因調査] → ログ分析、関係者ヒアリング
    ↓
[再発防止策策定] → 対策実施、教育・周知
    ↓
[報告書作成] → 完了報告
```

> **根拠**: IPA p.50-52

### 12.3 機密情報漏洩時の対応

1. **即時対応**: 当該ツールの利用一時停止
2. **影響範囲特定**: 漏洩したデータの特定、影響者の洗い出し
3. **顧客報告**: 顧客データの場合は速やかに顧客へ報告
4. **当局報告**: 個人情報漏洩の場合は個人情報保護委員会へ報告検討
5. **再発防止**: 原因分析、対策実施

### 12.4 再発防止策

- 原因分析（5 Whys、RCA等）
- 対策の策定・実施
- 教育・周知の実施
- ガイドライン・手順の改定（必要に応じて）

- 社内チャットでのAI関連情報共有
- 月次ニュースレター（任意）
